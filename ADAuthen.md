    # WorkSync - Active Directory Authentication & Integration Guide

เอกสารฉบับนี้สรุปโครงสร้างทางเทคนิค วิธีการ วิธีการตรวจสอบ และข้อควรระวังในการเชื่อมโยงการเข้าสู่ระบบผ่าน Active Directory (AD) เข้ากับโครงการ WorkSync เพื่อใช้เป็นเอกสารอ้างอิงและประสานงานร่วมกับทีมพัฒนาหรือผู้ควบคุมระบบ (Network/System Admin) ต่อไป

---

## 1. ข้อมูลการเชื่อมต่อ Target Endpoint (API Gateway)

ระบบยิงคำขอล็อกอิน (LDAP Bind) ไปยังปลายทาง Gateway ดังนี้:
- **Method**: `POST`
- **URL**: `http://192.168.12.8:3100/api/v2/login` (หรือใช้ตาม Env `AD_GATEWAY_URL`)

---

## 2. Request Payload (JSON Format)

รูปแบบการส่งชุดข้อมูลใน Request Body ต้องจัดส่งในลักษณะ JSON:

```json
{
  "app_id": "ProRegis",
  "secret_key": "d69f9e5a88e734c56e2978a63bf720c22635a9c0c32b5e2a2205510657e4e138",
  "username": "your_ad_username",
  "password": "your_ad_password",
  "timestamp": "2026-06-25T14:10:49Z"
}
```

### รายละเอียดฟิลด์ข้อมูล:
- **`app_id`**: ค่าคงที่ของแอปพลิเคชันปลายทาง เช่น `"ProRegis"`
- **`secret_key`**: คีย์ลับของแอปสำหรับขอสิทธิ์เข้าใช้งาน AD
- **`username`**: ชื่อผู้ใช้ AD ของบริษัท (ระวัง: ห้ามนำหน้าด้วยเครื่องหมาย `@`)
- **`password`**: รหัสผ่าน AD
- **`timestamp`**: วันเวลา ณ ขณะที่ยิง API (ตามกฎเกณฑ์พิเศษในข้อที่ 3)

---

## 3. ข้อควรระวังและกฎเกณฑ์เรื่องเวลา (Timestamp Verification)

ระบบของ AD Agent มีเงื่อนไขการตรวจสอบนาฬิกาที่เข้มงวดมาก:

1. **ตรวจสอบความสดใหม่ไม่เกิน 5 นาที (300,000 ms)**: หากผลต่างระหว่างเวลาที่ส่งไปกับเวลาในเครื่อง AD Agent ต่างกันเกิน 5 นาที ระบบจะปฏิเสธคำขอและตอบกลับเป็น `Expired` ทันที
2. **ฟอร์แมตเวลาประเทศไทยปิดท้ายด้วย Z (UTC+7 Local Time in Z Format)**:
   - ระบบตัวรับของ AD จะอ่านและเปรียบเทียบค่าโดยใช้เวลาท้องถิ่นประเทศไทย (+07:00) แต่ต้องการฟอร์แมตปิดท้ายด้วย **`Z`** และ**ต้องไม่มีเศษทศนิยม (มิลลิวินาที)**
   - **รูปแบบที่ถูกต้อง**: `YYYY-MM-DDTHH:mm:ssZ`
   - **ตัวอย่างเช่น**:
     - เวลาประเทศไทยปัจจุบันขณะยิง: **`11:15:49`**
     - ค่าที่ส่งไปใน JSON: **`2026-06-24T11:15:49Z`**
   - *หมายเหตุ: ห้ามส่งเป็นเวลาสากล UTC ที่ไม่ได้บวกเจ็ดชั่วโมง หรือส่งที่มีเศษทศนิยม เช่น 2026-06-24T04:15:49.123Z (จะทำให้ตัวรับแจ้งว่า Expired)*

---

## 4. โครงสร้างการทำงาน และระบบตรวจสอบสิทธิ์ผ่าน AD (Authentication Flow)

เพื่อความปลอดภัยสูงสุดและป้องกันการรั่วไหลของข้อมูลบัญชีผู้ใช้เมื่อมีการเปลี่ยนรหัสผ่านบน AD ระบบ**ยกเลิกการใช้ระบบแคชรหัสผ่าน (No Password Caching)** โดยมีขั้นตอนดังนี้:

1. **ขั้นตอนการเช็กข้อมูลผู้ใช้**:
   - เมื่อผู้ใช้เข้าสู่ระบบ ระบบจะค้นหาบัญชีผู้ใช้ในฐานข้อมูลหลักก่อน หากไม่มี หรือไม่ได้เปิดใช้งาน **"ใช้ Active Directory Authentication" (`isAdAuth = true`)** ระบบจะไม่อนุญาตให้ล็อกอินในโหมด AD
2. **การตรวจสอบสิทธิ์ตรงไปยัง AD Gateway (Direct Verification)**:
   - ทุกครั้งที่มีการระบุรหัสผ่าน AD (และไม่ใช่รหัสผ่านที่เป็น PIN Code) ระบบจะ**ยิงส่งข้อมูลคำขอจริงตรงไปยัง AD Gateway เสมอ** เพื่อตรวจสอบสิทธิ์การเข้าใช้งาน ณ ปัจจุบัน
   - **ไม่มีการนำรหัสผ่าน AD ไปแฮชและจัดเก็บลงในฐานข้อมูลหลัก (Local DB)** เพื่อป้องกันสิทธิ์ตกค้างหรือความเสี่ยงด้านความปลอดภัย (Credential Leaks) กรณีที่ผู้ใช้เปลี่ยนรหัสผ่านบน AD หลักแล้ว แต่แคชในฐานข้อมูลหลักยังเป็นรหัสเก่าอยู่
3. **ระบบ Bypass ด้วย PIN Code (PIN Bypass)**:
   - หากผู้ใช้ระบุรหัสผ่านที่เป็น PIN Code (ตัวเลข 6 หลักที่ตั้งค่าไว้ล่วงหน้าบนโปรไฟล์ผู้ใช้) ระบบจะตรวจสอบกับฟิลด์ `pinCode` ใน Local DB โดยตรงเพื่อความรวดเร็วในการทำงาน (ถ้าตั้งค่าไว้)

---

## 5. การตรวจสอบความปลอดภัยและ IP Filtering
- ฝั่ง AD Gateway เซิร์ฟเวอร์จะมีการทำ IP Filtering โดยเทียบกับไฟล์คอนฟิก `registry.json`
- ตรวจสอบให้มั่นใจว่า IP ของเครื่องต้นทางฝั่ง Client (เช่น IP ของเซิร์ฟเวอร์ VPS Hostinger `157.173.219.153` เป็นต้น) ได้รับการแอดเข้าไปในทะเบียนอนุญาตของ AD Agent เรียบร้อยแล้ว

---

## 6. คอนฟิกูเรชัน Environment Variables (.env)
- **`AD_GATEWAY_URL`**: URL ไปยังจุดรับ Authen (ค่าเริ่มต้น: `http://192.168.12.8:3100/api/v2/login`)
- **`AD_APP_ID`**: ตัวระบุแอปสำหรับ AD (ค่าเริ่มต้น: `ProRegis`)
- **`AD_SECRET_KEY`**: คีย์ลับเชื่อมต่อ (ค่าเริ่มต้น: `d69f9e5a88e734c56e2978a63bf720c22635a9c0c32b5e2a2205510657e4e138`)
